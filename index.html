My story

It took me WAYY more time that I was expecting to refine this blog page. But I guess this is the way to practice... yo have to experience some frustration. Specifically, I coudn't make Imager.js to work. OF course I will try again... sooner or later â˜º.

I used Grunt to resize the pictures, I made 4 version of each: 640 and 800 px wide, both in 1X and 2X pixel density. I found this way to manage the pictures unnecessary long, verbose and error prone, tough. I think sooner or later some smarter way will be implemented. I really hope so.

And by the way, the code provided in the Grountfile.js is wrong!! IT took me a while to find the problem... the commas is the suffix description must be double commas!! Instead there's single commas and the code doesn't work. Now, my Grountfile can make 4 version of the same picture in one passage, like is suppose to work!

I was exciting to test the complex media query with different resolution, pixel density and a fallback image! Especially to test the actual responsiveness with mobile devices!

I even implemented some basic type of art-direction! The horses image for the low resolution version is cropped!

Among the many things I've realized, is that the order of the source queries matter! The first resolution that is found, wins, so, I have to chance my code accordingly, by query first the higher resolution, and then the lower ones.

ON the bottom of the page, I put the links and the logos for the social medias. The links didn't look good, tough: in blue color and underlined. So, I modify the CSS code and make it look better.

To improve the pictures layout, I put a captionon the bottom, I centered it and make the font smaller. Besides, I round the image corners.

Future work: get more efficient in writing code for dinamically resizing pictures. I found there's a bunch of wervices on the web that can make the task way less daunting, but they are not easy to understand and implement. Besides, some of them are paid services, and they need to store my pictures in their services... not in ideal solution.

Final test: all the pictures resize correctly. The page size for viewport resolution below 640px is 0.41 MB, wich is great. It also means that all the fallback pictures are chosen correctly. At resolutions between 640 and 800px wide (DPR 2x) the pagebyte is 1.09 MB, with retina resolution pictures. At resolution higher than 800 px wide (DPR 2x) the pagebyte is 1.79 MB with retina images.

Lastly, the code was verified and amended until both the HTML and the CSS code where free of errors. I didn't wuite understand the itemprop error part, anyway, but I fixed it too! :-)
